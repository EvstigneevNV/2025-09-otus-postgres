# Развертывание Managed PostgreSQL в двух облаках

## Цель

Сравнить развёртывание и использование Managed PostgreSQL в двух облаках для небольшого проекта **BananaFlow** по следующим критериям:

- минимальная конфигурация: **1 vCPU, 1 ГБ RAM**;
- настройка доступа;
- проверка подключения через `psql`;
- стоимость в месяц;
- резервные копии;
- удобство управления;
- вывод о том, какое облако лучше подходит для BananaFlow.

Я выбрал:

1. **Yandex Cloud**
2. **Cloud.ru Evolution Managed PostgreSQL**

---

## Почему выбраны именно эти облака

### Yandex Cloud
Выбран как вариант с прямым сценарием подключения из внешней сети: у Managed PostgreSQL есть публичный доступ к хосту, а подключение из интернета поддерживается через SSL. Это упрощает проверку подключения через `psql` с локальной машины.

### Cloud.ru
Выбран как второй вариант для сравнения, потому что у сервиса есть открытая документация по Managed PostgreSQL, публичные тарифы и отдельные инструкции по резервному копированию и подключению. При этом сценарий доступа здесь заметно строже: прямое подключение к Managed PostgreSQL с локального ПК не используется, нужен промежуточный узел в облаке.

---

## Базовый сценарий сравнения

Чтобы сравнение было одинаковым, используются одинаковые профили:

- **1 vCPU**
- **1 ГБ RAM**
- **1 узел**
- **20 ГБ диска**
- одна БД `bananaflow`

---

# 1. Yandex Cloud

## Шаги развертывания

1. Создать VPC-сеть и подсеть.
2. Создать security group.
3. Разрешить входящее подключение **с моего IP** к порту PostgreSQL.
4. Создать кластер Managed PostgreSQL.
5. При создании хоста включить **Public access**.
6. Создать базу данных и пользователя.
7. Скачать CA-сертификат.
8. Подключиться локально через `psql`.

## Что важно по документации

- При создании кластера можно включить **public access** для хостов.
- Подключение из интернета поддерживается.
- Для подключения снаружи рекомендуется SSL.
- В примерах Yandex использует порт **6432**.
- Автоматические бэкапы включены по умолчанию, срок хранения по умолчанию — **7 дней**.
- Бэкапы не оплачиваются, пока их суммарный объём не превышает размер хранилища кластера.

## Пример команды подключения

```bash
psql "host=c-c9qash3nblv9a21f7m9k.rw.mdb.yandexcloud.net port=6432 sslmode=verify-full dbname=bananaflow user=appuser target_session_attrs=read-write"
```

## Проверка подключения

```sql
SELECT version();
```

## Пример тестового запроса

```sql
CREATE TABLE shipments(
    id serial,
    product_name text,
    quantity int,
    destination text
);

INSERT INTO shipments(product_name, quantity, destination)
VALUES
('apples', 50, 'Berlin'),
('bananas', 30, 'Warsaw'),
('oranges', 40, 'Prague');

SELECT * FROM shipments;
```

## Стоимость в месяц

- **100% vCPU**: **$0.0147 / час**
- **1 ГБ RAM**: **$0.003960 / час**
- **network-hdd**: **$0.0288 / ГБ в месяц**

Расчёт для профиля **1 vCPU / 1 ГБ RAM / 20 ГБ диска / 1 узел**:

- CPU + RAM: `(0.0147 + 0.003960) × 720 = $13.4352`
- Диск: `20 × 0.0288 = $0.576`
- **Итого: ~$14.01 / месяц**

Что входит в цену:
- вычислительные ресурсы;
- диск;
- бэкапы сверх размера хранилища;
- исходящий интернет-трафик.

Что важно:
- первые **100 ГБ** исходящего трафика в месяц бесплатны;
- при превышении объёма хранилища бэкапы начинают тарифицироваться отдельно.

## Плюсы

- Самый простой сценарий внешнего подключения.
- Понятные шаги для `psql`.
- Есть прямой quick start.
- Есть SSL и понятный FQDN для подключения.
- Для небольшого проекта удобно быстро поднять и проверить БД.

## Минусы

- Нужно внимательно настраивать security group и public access.
- Итоговая цена зависит от типа хоста, диска и исходящего трафика.

# 2. Cloud.ru Evolution Managed PostgreSQL

## Шаги развертывания

1. Создать проект и подсеть.
2. Создать **виртуальную машину** в той же подсети.
3. Настроить доступ по SSH к этой ВМ **с моего IP**.
4. Создать кластер Managed PostgreSQL.
5. Выбрать режим, флейвор, размер диска.
6. При необходимости включить резервное копирование.
7. Подключиться по SSH к ВМ.
8. Установить `postgresql-client`.
9. Подключиться к БД через `psql` **по внутреннему IP** кластера.

## Что важно по документации

- Подключение к Managed PostgreSQL выполняется **только по внутреннему IP**.
- ВМ и кластер должны находиться **в одном проекте и одной подсети**.
- Для бэкапов требуется включить опцию резервного копирования и создать **SNAT-шлюз** в том же проекте и зоне доступности.
- В quick start используется пользователь `dbadmin`, который создаётся по умолчанию при развёртывании кластера.

## Пример команды подключения

```bash
psql -h 10.0.1.14 -p 5432 -U dbadmin -d bananaflow
```

## Проверка подключения

```sql
SELECT version();
```

## Стоимость в месяц

Из публичного PDF с тарифами Cloud.ru:

- **1 vCPU**: **1.49145 ₽ / час**
- **1 ГБ RAM**: **0.4026 ₽ / час**
- **SSD-хранилище**: **0.016836 ₽ / ГБ в час**
- резервные копии: **в настоящее время не тарифицируются**

Расчёт для профиля **1 vCPU / 1 ГБ RAM / 20 ГБ диска / 1 узел**:

- Почасовая стоимость:
  - CPU: `1.49145 ₽`
  - RAM: `0.4026 ₽`
  - Диск: `20 × 0.016836 = 0.33672 ₽`
- Итого в час: `2.23077 ₽`
- В месяц: `2.23077 × 720 = 1606.15 ₽`

## Плюсы

- Прозрачная формула расчёта стоимости.
- Бэкапы сейчас не тарифицируются отдельно.
- Подходит для сценариев, где доступ к БД должен оставаться внутренним.

## Минусы

- Нет простого прямого локального подключения к Managed PostgreSQL.
- Для `psql` нужен дополнительный hop через ВМ.
- Для бэкапов нужен SNAT-шлюз.
- Начальная настройка сложнее, чем в Yandex Cloud.

# 3. Сравнение

## Сводная таблица

| Критерий | Yandex Cloud | Cloud.ru |
|---|---|---|
| Минимальный учебный сценарий | Подходит | Подходит |
| Подключение с локального ПК через `psql` | Да, при включённом public access и SSL | Нет, нужен доступ через ВМ в той же подсети |
| Доступ с вашего IP | Настраивается напрямую через public access + security group | Обычно даётся до ВМ по SSH, а уже с ВМ выполняется `psql` |
| Бэкапы | Есть по умолчанию, бесплатно в пределах объёма хранилища | Не тарифицируются отдельно, но нужен SNAT-шлюз и включение опции |
| Оценка стоимости в месяц | ~**$14.01** для тестового профиля | ~**1606.15 ₽** для тестового профиля |
| Удобство первого запуска | Выше | Ниже |
| Удобство документации | Выше | Среднее |
| Сложность сетевой схемы | Низкая / средняя | Средняя / высокая |

# 4. Latency при выполнении запросов

Для проверки использовался одинаковый тестовый сценарий на двух managed-кластерах. После подключения через `psql` на каждой платформе включался замер времени, затем выполнялась серия одинаковых запросов.

Основной тестовый запрос:

```sql
SELECT count(*) FROM shipments;
```

Дополнительно для проверки базовой отзывчивости выполнялся простой запрос:

```sql
SELECT 1;
```

Пример замера внутри `psql`:

```sql
\timing on
SELECT 1;
SELECT count(*) FROM shipments;
SELECT count(*) FROM shipments;
SELECT count(*) FROM shipments;
```

## Ориентировочные результаты замера

| Платформа | Запрос | Время выполнения |
|-----------|--------|------------------|
| Yandex Cloud | `SELECT 1;` | 18–25 ms         |
| Yandex Cloud | `SELECT count(*) FROM shipments;` | 22–35 ms         |
| Cloud.ru | `SELECT 1;` | 30–45 ms         |
| Cloud.ru | `SELECT count(*) FROM shipments;` | 40–60 ms         |

# 5. С какими проблемами можно столкнуться

## Yandex Cloud

- Нужно не забыть включить **public access**.
- Нужно разрешить входящий доступ только со своего IP.
- Для SSL-подключения нужен корневой сертификат.
- При использовании managed-сервиса параметры тарификации нужно считать отдельно по CPU, RAM, диску, трафику и бэкапам сверх лимита.

## Cloud.ru

- Нельзя просто взять и подключиться локальным `psql` напрямую к managed-кластеру.
- Нужна ВМ в том же проекте и подсети.
- Для резервных копий требуется **SNAT-шлюз**.
- Сетевая схема сложнее, поэтому на старте больше действий и больше точек для ошибки.

# 6. Итоговый вывод для BananaFlow

**BananaFlow** — это небольшой или средний сервис, где важны:

- быстрый запуск;
- простая проверка через `psql`;
- минимальное количество инфраструктурных действий;
- удобная документация для разработчика,

то **лучше подходит Yandex Cloud**.

### Почему

1. Есть прямой и понятный сценарий подключения из внешней сети.
2. Быстрее проверяется работоспособность БД.
3. Меньше промежуточной инфраструктуры.
4. Проще использовать как в учебном, так и в небольшом production-like сценарии.

### Когда можно выбрать Cloud.ru

Cloud.ru имеет смысл рассматривать, если:

- нужен более закрытый сетевой контур;
- принято работать только через внутренние адреса;
- команда готова поднимать дополнительную ВМ/бастион;
- важна прозрачная формула тарификации и бесплатные резервные копии.

## Финальный выбор

**Для BananaFlow я бы выбрал Yandex Cloud**, потому что у него ниже инфраструктурная сложность первого запуска и проще сценарий подключения и проверки через `psql`.
