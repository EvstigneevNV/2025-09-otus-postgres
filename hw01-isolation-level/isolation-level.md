# Отчёт по работе с транзакциями и уровнями изоляции

## 1. Работа с транзакциями

- В обеих сессиях был отключён **autocommit**.
- В первой сессии была создана таблица `shipments` и добавлены данные:

```sql
create table shipments(
    id serial,
    product_name text,
    quantity int,
    destination text
);

insert into shipments(product_name, quantity, destination)
values ('bananas', 1000, 'Europe');

insert into shipments(product_name, quantity, destination)
values ('coffee', 500, 'USA');

commit;
```

- Теперь в таблице есть две записи: про бананы и кофе.

---

## 2. Изучение уровней изоляции

1. Проверили текущий уровень изоляции:

```sql
show transaction isolation level;
```

Результат: **read committed** (уровень по умолчанию в PostgreSQL).

2. Начали новые транзакции в обеих сессиях.
3. В первой сессии добавили запись:

```sql
insert into shipments(product_name, quantity, destination)
values ('sugar', 300, 'Asia');
```

4. Во второй сессии выполнили:

```sql
select * from shipments;
```

- **Наблюдение:** новой записи (`sugar`) **не видно**, так как транзакция в первой сессии ещё не зафиксирована (`commit` не выполнен).
- После выполнения `commit;` в первой сессии и повторного `select` во второй сессии запись **стала видна**.
- **Почему:** уровень `read committed` гарантирует, что в каждой новой команде `select` во второй сессии будут видны только те изменения, которые уже зафиксированы (committed) другими транзакциями.

---

## 3. Эксперименты с уровнем изоляции Repeatable Read

1. В обеих сессиях установили уровень изоляции:

```sql
set transaction isolation level repeatable read;
```

2. В первой сессии добавили запись:

```sql
insert into shipments(product_name, quantity, destination)
values ('bananas', 2000, 'Africa');
```

3. Во второй сессии выполнили:

```sql
select * from shipments;
```

- **Наблюдение:** новой записи **не видно**, так как транзакция первой сессии ещё не завершена.
- После `commit;` в первой сессии — во второй сессии при повторном `select` новая запись всё равно **не появилась**.
- **Почему:** при уровне `repeatable read` каждая транзакция «видит» снимок базы данных на момент её начала и не видит изменения, сделанные другими параллельными транзакциями.
- После завершения второй транзакции и запуска новой `select * from shipments` — запись **стала видна**.
- **Вывод:** уровень `repeatable read` защищает от так называемых «phantom reads» (фантомных чтений), но при этом изменения других транзакций становятся видны только после завершения текущей транзакции.

---

## Итоговые выводы

- **Read committed:** изменения становятся видны другим сессиям только после фиксации (`commit`), и каждый новый запрос читает актуальное состояние БД.
- **Repeatable read:** транзакция работает с одним «снимком» данных, поэтому даже после `commit` в других сессиях внутри текущей транзакции новые изменения **не видны**, пока сама транзакция не завершится.
- Алексей подтвердил на практике, как уровни изоляции влияют на видимость данных между параллельными сессиями.
